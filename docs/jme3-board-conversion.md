# jME3 board/bounty conversion (Phase 5, step 6)

One-time fork→non-fork conversion of the 364 binary **data** assets that have no XML
source and so cannot be regenerated by the build the way `model.dat` is:

| Asset | Count | On disk today | Loaded by |
|---|---|---|---|
| game boards | 165 | `data/boards/<town>/<n>/*.board` | `BoardManager` (server) — `BoardFile.loadFrom`, `BoardManager.java:44` |
| menu backdrops | 3 | `assets/rsrc/menu/<town>/town.board` | `TownView` / menu (same `BoardFile`) |
| bounty games | 196 | `assets/rsrc/bounties/.../*.game` | `OfficeManager` — `BinaryImporter.load`, ~`OfficeManager.java:118` |

All 364 are vendored jME-**fork** `BinaryExporter` `Savable` graphs (`com.jme.util.export.*`).
When the fork is removed (Phase 5) there is no reader for them, so they must be converted
once, offline, with the fork reader retained as a leaf tool — never as runtime code.

This document records (1) what is inside each file, (2) the proof that the on-disk format
can switch to **Narya streaming** with no new format and no jME3 `BinaryExporter`, (3) the
conversion tool and its full-corpus round-trip results, and (4) the runtime read-path change
needed at cutover (design only — the running load paths are not modified by this step).

## 1. What is inside a `.board`

`BoardFile implements com.jme.util.export.Savable`
(`client/shared/.../game/util/BoardFile.java`). Its `Savable` read/write persists:

```
String   name              // human-readable board name
String   creator           // author username, or "" for system boards
String[] scenarios         // scenario ids this board supports
boolean  privateBoard      // excluded from random match-made selection
int      players           // player count this board is built for
BangBoard board            // the terrain + environment (Savable graph below)
List<Piece> pieces         // props, markers, tracks, viewpoints, ...
// dataHash is DERIVED on read (MD5 of toData().toBytes()), not stored
```

`BangBoard extends com.threerings.io.SimpleStreamableObject implements ..., Savable`
(`game/data/BangBoard.java`). Its `Savable` capsule (and, identically, its Narya
auto-streamed non-transient fields) carry:

```
int      _width, _height                  // tile dimensions
byte[]   _heightfield                     // (4·w+1)·(4·h+1) elevation samples (the big array)
byte     _elevationUnitsPerTile
byte[]   _terrain, _shadows               // splat/shadow grids
float    _shadowIntensity
byte     _waterLevel;  int _waterColor;  float _waterAmplitude
float[]  _lightAzimuths, _lightElevations // NUM_LIGHTS=2 directional lights
int[]    _lightDiffuseColors, _lightAmbientColors
int      _skyHorizonColor, _skyOverheadColor;  float _skyFalloff
float    _windDirection, _windSpeed
int      _fogColor;  float _fogDensity;  int _gridColor
StreamableHashMap<String,byte[]> _patchMap  // per-tile terrain patches
// _teams + a dozen transient caches (_pterrain, _tstate, ...) are rebuilt by
// initTransientFields() and are NOT persisted in either format
```

`Piece` (`game/data/piece/Piece.java`) `implements Streamable, Cloneable, Savable, ...`,
base fields `pieceId, owner, lastActed, x, y, orientation, damage, scenId` plus subtype
fields. Concrete board pieces seen in the corpus: `Prop`, `Marker`, `Track`, `Viewpoint`,
`Teleporter` (a Prop), plus the rest of the hierarchy (`Bonus`, `Unit`, `Cow`, `Train`,
`Wendigo`, `Barricade`, `BigPiece`). Note `Prop.read` calls
`Prop.init(PropConfig.getConfig(type))`, so reconstituting a board requires the prop/unit/
bonus **config registry** to be populated from the staged `rsrc/.../*.txt` lists (see §3).

## 2. What is inside a `.game`

A bounty `.game` is a single `BangConfig` Savable graph (`game/data/BangConfig.java`):

```
BangConfig extends GameConfig (a streamed PlaceConfig) implements Savable
  boolean rated                  // inherited GameConfig field
  Type type        {TUTORIAL,PRACTICE,SALOON,BOUNTY}  // written as String
  Duration duration{PRACTICE,QUICK,NORMAL,LONG}        // written as String
  Speed speed      {FAST,NORMAL,SLOW}                  // written as String
  int minWeight;  boolean respawnUnits
  List<Round>     rounds    // Round{scenario, board, sdata; bdata NOT serialised}
  List<Player>    plist     // Player{bigShot, units, cards, startSpot, teamIdx, skill}
  List<Criterion> criteria  // bounty win conditions
```

`Round` and `Player` are `SimpleStreamableObject ... Savable` inner classes. `Criterion`
(`game/data/Criterion.java`) is `abstract ... extends SimpleStreamableObject implements
Savable`; the two concrete subtypes are `IntStatCriterion` and `RankCriterion`
(`bounty/data/`). The 196 corpus games use 0–4 criteria each.

## 3. Target format — proven: Narya streaming (no new format, no jME3 BinaryExporter)

The decisive fact is that **every class in both graphs is already a Narya
`com.threerings.io.Streamable`** — the exact mechanism by which boards and game configs
reach clients over the wire today:

| Class | Streamable proof |
|---|---|
| `BangBoard` | `extends SimpleStreamableObject` (Streamable); already streamed as the wire board (engine-notes "Boards" — the 201×201 heightfield is its largest streamed array) |
| `Piece` (+ all subtypes) | `implements Streamable`; pieces are `DSet.Entry` streamed in `BangObject` |
| `BoardData` (board+pieces) | `implements Savable, Streamable` — the dual-typed wrapper |
| `BangConfig` | `extends GameConfig` → a `PlaceConfig`, which Narya streams on every place creation |
| `BangConfig.Round/Player`, `Criterion` (+subtypes) | all `extends SimpleStreamableObject` |

So the on-disk format can switch from fork-`BinaryExporter` to **Narya streaming** with
**no new format invented and no jME3 dependency on the read path**. This was compared
against the two alternatives:

- **jME3 `BinaryExporter`** (what `tools/j3o-converter` does for models): viable, but it
  re-introduces a `Savable` contract on every board/config/piece class (`com.jme3.export.*`)
  and a jME3 dependency on the server's board-load path purely to read static data that the
  game already knows how to stream. Rejected: more code, more deps, no benefit here.
- **A neutral format (JSON/protobuf/custom)**: would require hand-writing and maintaining a
  schema for `BangBoard` + the whole `Piece` hierarchy + `BangConfig`/`Criterion`. Rejected:
  large surface, duplicates information the streaming layer already encodes.

Narya wins on all axes: zero new format, zero runtime fork/jME3 dependency, and it reuses
battle-tested serialization the game runs every session.

### On-disk layout written by the tool

- **`.nboard`** — the `BoardFile` header then the board graph, via one
  `com.threerings.io.ObjectOutputStream`:
  `UTF name · UTF creator · int players · boolean private · writeObject(scenarios:String[])
  · writeObject(board:BangBoard) · writeObject(pieces:ArrayList<Piece>)`.
  `dataHash` is derived on read exactly as the fork format does, so it is not stored.
- **`.ngame`** — a single `writeObject(config:BangConfig)`.

(Equivalently, a `.nboard` could stream a single `BoardData` plus the small header; the tool
streams the fields directly to keep the header next to the graph. Either is pure Narya.)

One nuance worth recording: the fork `Savable` form of `BangConfig` deliberately writes a
*subset* with legacy field names (`units`→"team", `plist`→"teams", and `bdata`/`ais`
skipped). The Narya form streams the live object's non-transient fields. For a one-time
conversion this is fine and in fact loss-free: we read the old file with the fork reader
(which fully populates the in-memory object), then stream that object; `bdata`/`ais` are
null/empty in on-disk bounty files anyway, and round-trip verification (§4) confirms every
load-bearing field survives.

## 4. The conversion tool and its results

`tools/board-converter` (module `tools:board-converter`, `mainClass
com.threerings.bang.tools.board.BoardConverter`). Additive leaf, modelled on
`tools/j3o-converter`: it is the only consumer that drags the fork export subsystem in for
offline use, and nothing depends on it. Dependencies: `client:shared` (game data classes +
retained fork reader via `jme`/`app`, and Narya as an `api` dep) and `runtimeOnly` depot
(Narya's default enum-streamer policy references `com.samskivert.depot.util.ByteEnum`; see
engine-notes "Streaming any enum touches depot").

Per file it: (1) reads the legacy file with the **retained fork reader**
(`BoardFile.loadFrom` / `BinaryImporter.load`); (2) re-serialises to `.nboard`/`.ngame` via
Narya into a **separate output dir** (no real asset is modified or deleted); (3) re-reads the
Narya output and asserts field parity against the original:

- boards: name, creator, players, private flag, scenarios, board width/height, heightfield
  length **and byte-for-byte hash**, and an order-independent census of every piece's
  `class@x,y:pieceId`.
- games: type/duration/speed/minWeight/respawn, every round's scenario|board|sdata, every
  player's bigShot|units|cards|startSpot|teamIdx|skill, and every criterion's type+description.

Runs fully headless (stands up `com.jme.util.DummyDisplaySystem`, as `j3o-converter` does).
`-Dappdir=<assets staging dir>` is required so `PropConfig`/`UnitConfig`/`BonusConfig` can
resolve piece configs from the staged `rsrc/.../*.txt` lists during the fork read.

### Full-corpus result — 364/364 PASS

Run (after `./gradlew assets:processResources` produced the staging tree):

```
./gradlew :tools:board-converter:run \
  --args="<repo>/data/boards <out>/data-boards --boards" -Dappdir=<repo>/assets/build/staging
./gradlew :tools:board-converter:run \
  --args="<repo>/assets/rsrc/menu <out>/menu --boards"  -Dappdir=...
./gradlew :tools:board-converter:run \
  --args="<repo>/assets/rsrc/bounties <out>/bounties --games" -Dappdir=...
```

| Corpus | Files | Round-trip parity |
|---|---|---|
| `data/boards/**/*.board` | 165 | **165 ok / 0 fail** |
| `assets/rsrc/menu/*/town.board` | 3 | **3 ok / 0 fail** |
| `assets/rsrc/bounties/**/*.game` | 196 | **196 ok / 0 fail** |
| **Total** | **364** | **364 ok / 0 fail** |

No problem files. Boards span 30×30 to 50×50 with heightfields of 14,641–40,401 bytes and up
to 345 pieces; bounty games span 1+ rounds, up to 4 criteria.

**Note on `boom_town/town.board`:** the boom_town backdrop initially failed
(`NullPointerException` in `Prop.init` — `PropConfig.getConfig` returned null) because
`boom_town` is **not** a shipped town (root `build.gradle` `towns = [frontier_town,
indian_post]`), so the asset build never generates `props/boom_town/props.txt`. This is a
missing generated **list**, not a tool/format problem: generating boom_town's lists into the
staging artifact (`assets/update_lists build/staging boom_town`) makes it convert cleanly
(288 pieces). All three backdrops are in the 364/364 above. If boom_town is to be shipped,
its lists must be generated regardless of this step.

## 5. Runtime read-path change at cutover (design only — NOT implemented here)

At Phase 5 cutover the fork is gone, so the three load sites must read the Narya format. The
changes are small and local; no class needs a new serialization contract (they are already
Streamable):

1. **`BoardFile`** (`game/util/BoardFile.java`). Today `loadFrom(File/InputStream/byte[])`
   delegates to `BinaryImporter`; `saveTo` to `BinaryExporter`; the class also implements the
   fork `Savable` read/write. At cutover:
   - Replace the three `loadFrom` bodies with the `.nboard` Narya read (the `decodeBoard`
     logic in `BoardConverter`): a `com.threerings.io.ObjectInputStream` reading
     `name/creator/players/private/scenarios/board/pieces`. `dataHash` stays derived.
   - Replace `saveTo` with the matching `encodeBoard` Narya write (the **editor** save path).
   - Delete the `Savable` `read`/`write`/`getClassTag` and the `com.jme.util.export.*`
     imports. `BoardData.fromBytes`/`toBytes` similarly switch to Narya (they are already
     `Streamable`, so `toBytes` becomes a plain `ObjectOutputStream.writeObject(this)` /
     `fromBytes` an `ObjectInputStream.readObject()`).
2. **`BoardManager`** (server). No structural change: it still walks `data/boards` and calls
   `BoardFile.loadFrom`. Only the on-disk extension changes (`.board`→`.nboard`); either keep
   the `.board` name with the new internal format (simplest — `mapBoards` already filters on
   `.endsWith(".board")`) or update the filter. Recommend keeping the `.board`/`.game`
   extensions and just changing the bytes inside, so paths, the editor, and `BountyConfig.
   getGamePath` are untouched.
3. **Bounty loading** (`OfficeManager`, ~line 118: `BinaryImporter.getInstance().load(...)`
   on the `.game` resource). Replace with `new ObjectInputStream(stream).readObject()` cast
   to `BangConfig`. `BountyConfig.getGamePath` and the resource lookup are unchanged.
   The `BountyGameEditor` save/load (`bounty/client/BountyGameEditor.java:422`,
   `BinaryImporter.load`) switches the same way.

Because the runtime classes are unchanged Streamables, the **migration is: run this tool over
the 364 files to produce the Narya bytes, drop them in place (replacing the old fork bytes),
then swap the ~4 reader/writer methods above to Narya.** No piece/board/config field needs a
jME3 `Savable` annotation, and the server's board path gains no jME3 dependency.
